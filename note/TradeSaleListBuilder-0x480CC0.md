# 교역소 판매목록 빌더 역분석 (fb29 후속)

대상: **한국어판 `cds_95.exe`** (Ver.1.2.0.0, MD5 `c2b292460f4c4fb582a78909efd2c047`,
경로 `대항해시대3/cds_95.exe`). ImageBase 0x400000.
`.text` 0x401000, `.rdata` 0x4C3000, `.data` 0x52F000.

> 주의: `cds95runfilejp/CDS95.EXE`는 **일본판 1.4.0.0**(MD5 824312671…)이라 주소가 다르다.
> 문화권 교역품 테이블(0x4DF0E0)·판매 빌더(0x480CC0)는 위 **한국어판 EXE 기준**으로만 맞는다.
> (일본판에서 0x4DF0E0 덤프하면 쓰레기값 → 빌드 확인 지표로 사용) [[cds95-live-addresses-differential-scan]]

## 함수 0x480CC0 = BuildSaleList (판매목록 빌더)

- fb28이 준 `0x480E80`은 **함수 시작이 아니라** 메인 루프(수입품 phase) 중간이다.
  진짜 함수 시작 = **0x480CC0** (앞 int3 패딩 0x480CBB×5).
- 시그니처(추정):
  ```c
  // __thiscall, ret 0xC (인자 3개)
  int BuildSaleList(Ctx* ecx, Rec16* out1, int* out2names, int* out3qty);
  ```
- **count-then-fill** 2패스. 호출부 0x481580:
  1. `out1=out2=out3=NULL`로 호출 → **개수 N** 반환 (버퍼 NULL이면 카운터만 증가).
  2. `malloc(N*0x10)`, `malloc(N*4)`, `malloc(N*4)` 후 실제 버퍼로 재호출 → 채움.
- `Ctx`(ecx) = 교역 컨텍스트 객체. 호출 사슬: 0x4819C9 ← 0x481580(교역 대화 핸들러,
  "미안하지만 자네에게 팔 물건은…" 문자열 0x532928) ← ecx=esi.
  - `Ctx+0x90` = 현재 도시 id (원산지 기본값)
  - `Ctx+0xB0` = 수입품 배열 base, `Ctx+0xB8` = 그 개수
  - `Ctx+0x80` = 도시 표시명(에러 경로에서 사용)

### out1[i] 레코드 (0x10 바이트)
| off | 내용 |
|---|---|
| +0 | 교역품 종류 index (0~69, `kTradeGoods`) |
| +4 | 표시명 문자열 "○○산□□" (빌더 **0x480890**) |
| +8 | 원산지 도시 id |
| +C | (미상) |

out2[i] = 이름 문자열(0x480890 결과), out3[i] = 수량(마지막 루프에서 0x4B5910 rng ×3/2 감쇠).

## 판매목록의 3개 출처(phase) — 확정
1. **문화권 공통품** (0x480D4F 루프, 헬퍼 `0x42A160`): `0x42A160`이 `0x4DF0E0[문화권]`(i32×5,-1종료)을
   읽어 목록 채움. **문화권 교역품 테이블 유일 xref = 0x42A176.** 문화권 인덱스 = `[0x4D14B0+idx*136 +0x1C]`.
   자기 도시 특산품과 같은 품목은 **중복 제외**(0x42A160 내 0x42A030 비교). 원산지 = `Ctx+0x90`(이 도시).
   → 밀·총 같은 제조/공통품.
2. **자기 특산품** (0x480DD9, 단일): 종류 = 도시struct+0x10, 공급 = struct+0x18. 원산지 = `Ctx+0x90`.
   `0x58BAB0[종류]==0`이면 skip(0x480DEF). → **이스탄불 골동품이 여기서 걸러짐.**
3. **연결 내륙도시 특산품** (0x480E4C 루프, ← 0x480E80 포함): `Ctx+0xB0`는 **인라인 리스트**
   (`add eax,0xB0`, 개수 `Ctx+0xB8`) — 보통 0~2개. 각 원소 = 연결 내륙도시 id.
   `0x429970(id)`→도시struct, `0x42A030`→특산품 종류, 원산지 = 그 도시. → "코르도바산 피혁" 류.
   이스탄불은 개수 0 → 그래서 밀·총만.

### 공통 게이트: `0x58BAB0[종류] != 0`
- `lea ecx,[종류*4 + 0x58BAB0]` 3회 등장. `0x58BAB0` = **런타임 교역품 데이터 포인터
  테이블**(종류별, BSS라 파일엔 0). 해당 종류 포인터가 null이면 그 교역품은 목록에서 제외.

## fb29(이스탄불=밀·총만) — 확정 (라이브 RPM 실측)
- 이스탄불 struct `0x588BE8`(=0x5863A8+112*92): +0x10 종류=**47 골동품**, +0x18 재고=**20**,
  +0x58 문화권=4(중근동), +0x0C 시세=104. → 재고 멀쩡함.
- **`0x58BAB0[47] = 0` (게이트 OFF)** 이 원인. `0x58BAB0[종류]`는 포인터가 아니라 **판매허용
  플래그(1/0)**. 밀[0]=1, 총[21]=1이라 팔리고 골동품[47]=0이라 안 팔림.
- 70종 스캔: 43켜짐/27꺼짐. 꺼진 건 쌀·옥수수·감자·후추·정향·계피·커피·차·카카오·담배·골동품·
  융단·유향·백단 등 **아시아·인도·아메리카 원거리 품목** → `0x58BAB0` = **지역발견/시대 언락 상태
  (동적)**. 진행하면 값이 바뀜.
- 결론: 정적 "문화권공통(0x4DF0E0) + 도시 spec필드" 모델의 **"+ spec필드"가 틀림**.
  실제 규칙 = **(자체 생산품 phase1 + 수입품 Ctx+0xB0) 를 `0x58BAB0[종류]!=0`로 게이트**.
  spec 필드는 직접 안 쓰이고, 게이트는 라이브로 읽어야 정확.

## 툴 적용 방향
1. **라이브 훅(현재 항구 정확)**: 항구 진입 시 빌더가 채운 `out1` 또는 `Ctx+0xB0`을 읽으면
   현재 방문 항구의 판매목록을 100% 정확히 얻는다. 단 빌더는 **per-port** — 방문 중 1곳만.
2. **전 도시 교역품 창**: 빌더 직접 사용 불가(per-port). 근사는 유지하되 `BuildGoods()`의
   "+ 도시 spec필드"를 버리고 **도시별 생산품 목록**(phase 1 소스 = 0x4A17C0/0x42A160 경로의
   도시별 접근 함수)으로 교체해야 이스탄불류 오탐 제거. → 다음 과제: 도시별 생산품 목록
   접근자(0x4A17C0가 Ctx→무엇을 반환하는지, 도시 id로 직접 얻는 경로) 확인.

## 참고 헬퍼
- `0x480890` : "원산지산품목" 표시명 생성(문자열 테이블 0x547150/0x543728/0x547148 등).
- `0x4A17C0` : Ctx→서브객체 접근자(74곳 호출, 게임 전역 accessor).
- `0x42A160` : 서브객체의 목록 개수 반환 / 버퍼 채움(오버로드적 사용).
- `0x429970`,`0x42A030`,`0x42A100` : id→객체, 객체→종류index, 객체→가격 계열.

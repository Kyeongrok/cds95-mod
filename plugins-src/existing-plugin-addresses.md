# 기존 플러그인 정적 분석 결과 (하드코딩 주소)

`CDS95Util/*.plugin`은 소스가 없지만, 컴파일된 바이너리 자체에 일본판 EXE용
하드코딩 주소가 그대로 들어있습니다. Cheat Engine 같은 동적 분석 없이도,
바이트 패턴만으로 상당 부분을 뽑아낼 수 있어서 그 결과를 기록해둡니다.

## 방법

모든 대상 플러그인이 버전을 판별할 때 다음과 같은 동일한 패턴을 씁니다:

```
3D <imm32>        ; CMP EAX, imm32
```

이때 `imm32`를 4바이트로 쪼개면 그 자체가 `major.minor.build.revision`입니다.
예를 들어 `3D 00 07 02 01`은 리틀엔디안으로 `0x01020700`이고,
바이트별로 읽으면 `01.02.07.00` = **Ver.1.2.7.0**입니다.

이 CMP 직후 분기(JNZ로 스킵)에서 실제 주소가 다음 두 가지 형태로 나타납니다:

- `C7 05 <addr32> <imm32>` → `MOV dword ptr [addr32], imm32` — 플러그인 내부 전역 변수에
  게임 주소(`imm32`)를 캐싱하는 형태. `addr32`는 플러그인 자신의 주소(무시), `imm32`가
  실제 게임 주소.
- `8B 3D/35/0D <addr32>` → `MOV EDI/ESI/ECX, [addr32]` — 게임의 절대 주소를 직접
  참조(역참조)하는 형태. 이때는 `addr32` 자체가 게임 주소.

`plugins-src/tools/scan-hardcoded-addresses.ps1`에 이 스캔 로직을 스크립트로 남겨뒀습니다.

## 확인된 결과

| 플러그인 | 대상 버전 | 발견된 주소 | 형태 |
|---|---|---|---|
| CollectionUtil | 1.2.7.0 | `0x0058DE50` | 단일 함수 주소 (창고 기능 후킹 대상 1개) |
| CollectionUtil | 1.4.0.0 | `0x005FE1C0` | 〃 |
| TradeUtil | 1.2.7.0 | `0x0058E250`, `0x0058DE2C`, `0x0058DDEC` | 3개 세트 (4바이트 간격 아님, 개별 캐싱) |
| TradeUtil | 1.4.0.0 | `0x0056B3B0`, `0x005FE19C`, `0x005FE15C` | 〃 |
| HotelUtil | 1.2.7.0 | `0x00571058`, `0x0057105C`, `0x00571060` | 4바이트 간격 연속 3개 → 구조체 헤더로 추정 (count + 포인터×2) |
| HotelUtil | 1.4.0.0 | `0x0057CA90`, `0x0057CA94`, `0x0057CA98` | 〃 |
| SaveUtil | - | 못 찾음 | 다른 비교/저장 패턴 사용 (이번 스캔 범위 밖) |
| CPUPatch | 1.2.7.0 / 1.4.0.0 | 버전 비교만 확인, 주소는 별도 분석 필요 | `PatchUtil` 기능을 통해 처리되는 것으로 보임 |

**중요**: `CollectionUtil`, `TradeUtil`, `HotelUtil`, `SaveUtil`, `CPUPatch` 5개 전체를
스캔했지만 **한국어판(Ver.1.2.0.0)에 대한 버전 분기는 단 하나도 없습니다.**
문서(`README.md`의 한국어판 호환성 표)에 적힌 내용이 바이너리 레벨에서도 확인된 것입니다.

## 한계 — 왜 이 주소를 바로 못 쓰는가

이 저장소에는 한국어판 `CDS95.exe`(Ver.1.2.0.0)만 있고 일본판(1.2.7.0/1.4.0.0)은 없습니다.
위 주소들이 일본판 EXE에서 실제로 무엇을 가리키는지(함수 시그니처, 인자, 데이터 구조)를
디스어셈블로 검증할 방법이 없기 때문에, 이 주소를 한국어판에 그대로 대입할 수 없습니다.
버전이 다르면 컴파일된 코드 배치가 달라질 수 있어 위험합니다.

다만 다음과 같은 구조적 힌트는 확보한 상태입니다:

- CollectionUtil의 창고 기능은 후킹 대상이 **함수 1개**이며, 두 일본판 버전 모두
  `0x0058xxxx`~`0x0060xxxx` 코드 영역대에 위치합니다.
- HotelUtil은 **연속된 3개의 DWORD로 이루어진 작은 구조체**(아마 카운트 + 포인터 2개)를
  참조합니다.
- TradeUtil은 **서로 떨어진 위치의 주소 3개**를 개별적으로 캐싱합니다.

   위 주소들을 Ghidra로 직접 디스어셈블하면, 함수 시그니처/구조체 레이아웃을 완전히

   가장 정확하지만 일본판 EXE 입수가 전제 조건입니다.

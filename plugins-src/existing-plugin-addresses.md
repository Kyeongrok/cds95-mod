# 기존 플러그인 정적 분석 결과 (하드코딩 주소)

`CDS95Util/*.plugin`은 소스가 없지만, 컴파일된 바이너리 자체에 일본판 EXE용
하드코딩 주소가 그대로 들어있습니다. Cheat Engine 같은 동적 분석 없이도,
바이트 패턴만으로 상당 부분을 뽑아낼 수 있어서 그 결과를 기록해둡니다.

## 방법

모든 대상 플러그인이 버전을 판별할 때 다음과 같은 동일한 패턴을 씁니다:

```
3D <imm32>        ; CMP EAX, imm32
```

이때 `imm32`를 4바이트로 쪼개면 그 자체가 `major.minor.build.revision`입니다.
예를 들어 `3D 00 07 02 01`은 리틀엔디안으로 `0x01020700`이고,
바이트별로 읽으면 `01.02.07.00` = **Ver.1.2.7.0**입니다.

이 CMP 직후 분기(JNZ로 스킵)에서 실제 주소가 다음 두 가지 형태로 나타납니다:

- `C7 05 <addr32> <imm32>` → `MOV dword ptr [addr32], imm32` — 플러그인 내부 전역 변수에
  게임 주소(`imm32`)를 캐싱하는 형태. `addr32`는 플러그인 자신의 주소(무시), `imm32`가
  실제 게임 주소.
- `8B 3D/35/0D <addr32>` → `MOV EDI/ESI/ECX, [addr32]` — 게임의 절대 주소를 직접
  참조(역참조)하는 형태. 이때는 `addr32` 자체가 게임 주소.

`plugins-src/tools/scan-hardcoded-addresses.ps1`에 이 스캔 로직을 스크립트로 남겨뒀습니다.

## 일본판 Ver.1.4.0.0 EXE로 교차 검증 (완료)

일본판 `CDS95.exe`(Ver.1.4.0.0, MD5 `824312671ca0a7f25fa53ea7631ae093`)를 로컬에 확보해서
위 주소들을 실제로 검증했습니다. (이 EXE는 저작권이 있는 게임 실행 파일이라 리포에는
커밋하지 않고 `.gitignore` 처리된 `cds95runfilejp/` 폴더에만 로컬로 둡니다.)

PE 섹션 테이블을 직접 파싱해서 각 VA(Virtual Address)가 어느 섹션의 어느 오프셋에
해당하는지 계산한 결과, **1.4.0.0 버전의 주소 7개(CollectionUtil 1개, TradeUtil 3개,
HotelUtil 3개) 전부가 `.data` 섹션 중에서도 파일에는 실제 데이터가 없고 `VirtualSize`만
잡혀 있는 영역**(로드 시 0으로 초기화되는, PE의 사실상 BSS 영역)에 위치했습니다.

```
.text  VA=0x00001000 VirtualSize=0x000BA0A5 RawSize=0x000BB000
.rdata VA=0x000BC000 VirtualSize=0x00059C0E RawSize=0x0005A000
.data  VA=0x00116000 VirtualSize=0x000F7F48 RawSize=0x0004B000   <- 여기, RawSize보다 VirtualSize가 훨씬 큼
.rsrc  VA=0x0020E000 VirtualSize=0x00002178 RawSize=0x00003000
```

`.data`의 `RawSize`(0x4B000)보다 `VirtualSize`(0xF7F48)가 훨씬 커서, 파일 오프셋
0x116000+0x4B000=0x161000 이후 영역은 파일에 실체가 없고 프로세스가 로드될 때 0으로
채워지는 메모리입니다. 우리가 찾은 7개 주소 전부 이 범위(0x161000~0x20E000, VA 기준으로는
대략 0x00477000~0x0060E000) 안에 들어갑니다.

**결론: 이 7개 주소는 함수가 아니라 게임이 런타임에 채우는 데이터(카운터/포인터 등)였습니다.**
즉 플러그인들은 MinHook으로 이 주소의 "함수"를 후킹하는 게 아니라, 이 주소에 있는
**게임의 살아있는 메모리 값을 직접 읽고 쓰는 방식**으로 동작합니다 — Cheat Engine으로
메모리 스캔해서 찾는 것과 정확히 같은 종류의 대상이라는 뜻입니다. (실제 후킹은 아마
UI/메뉴 연동 부분에서 별도로 일어날 것으로 보이며, 이번 스캔 범위 밖입니다.)

이 발견은 한국어판에서 같은 것을 찾을 때도 그대로 적용됩니다: Ghidra로 "함수"를 찾으려
하지 말고, Cheat Engine으로 **살아있는 값**(창고 아이템 개수, 호텔 데이터 카운트 등)을
찾는 데 집중하면 됩니다. `plugins-src/README.md`의 절차와 정확히 일치하는 방향입니다.

## 게임 코드에서의 참조 횟수 (XREF, Ghidra 없이 원시 바이트 스캔으로 대체)

Ghidra 같은 전용 디스어셈블러 없이, `.text` 섹션 전체(파일 오프셋 0x1000~0xBC000)에서
7개 주소값과 정확히 일치하는 4바이트 리틀엔디안 시퀀스를 찾아 "이 주소를 참조하는
코드가 실제로 몇 군데인지" 세어봤습니다 (간접/계산된 주소 지정은 이 방법으로 못 잡는다는
한계는 있음).

| 주소 | 게임 코드 참조 횟수 | 해석 |
|---|---|---|
| CollectionUtil `0x005FE1C0` | **0건** | 게임 원본 코드가 전혀 안 씀. "무제한 창고"는 원래 게임에 없던 완전히 새로운 기능이라, 이미지 안의 안 쓰는 빈 공간(BSS 꼬리)을 플러그인이 자기 전용 저장소로 재활용한 것으로 추정. 즉 이 정확한 주소 자체는 한국어판에서 그대로 찾아야 할 의미 있는 대상이 아니라, 그냥 "안전하게 재활용 가능한 빈 공간" 하나였을 뿐일 가능성이 높음. |
| TradeUtil `0x0056B3B0` | 2건 | 게임 코드가 거의 안 씀(초기화 구간으로 추정) |
| TradeUtil `0x005FE19C` | **75건** | 게임 전역에서 활발히 참조되는 진짜 라이브 상태값(가격/재고 등으로 추정) |
| TradeUtil `0x005FE15C` | 10건 | 마찬가지로 게임이 실제로 쓰는 값 |
| HotelUtil `0x0057CA90` | 50건 | 진짜 라이브 상태값 |
| HotelUtil `0x0057CA94` | 26건 | 〃 |
| HotelUtil `0x0057CA98` | 12건 | 〃 |

HotelUtil의 3개 주소는 같은 명령어 근처에서 함께 등장하는 경우가 많습니다
(예: `A1 90CA5700 ... 8B15 94CA5700`처럼 바로 이어서 두 주소를 같이 읽음) — 애초 추정대로
**count + 포인터×2 형태의 작은 구조체**일 가능성이 높다는 게 참조 패턴으로도 뒷받침됩니다.

## 한국어판 작업에 대한 시사점 (갱신)

- **TradeUtil/HotelUtil처럼 게임 코드가 실제로 참조하는 주소**는 한국어판에서도 반드시
  대응하는 "진짜" 데이터가 있을 것이므로, Cheat Engine으로 라이브 값을 찾는 접근이 그대로
  유효합니다.
- **CollectionUtil처럼 참조 횟수가 0인 주소**는 애초에 "게임의 특정 데이터"가 아니라
  "빈 공간 재활용"이었을 수 있습니다. 그렇다면 한국어판 `CollectionUtilKR`을 만들 때 게임
  이미지 안에서 굳이 비슷한 빈 공간을 찾아 쓸 필요 없이, **플러그인 자신의 힙/정적 메모리에
  직접 저장소를 만드는 게 더 간단하고 안전합니다** (`plugins-src/README.md`에서 이미
  "표준 C 파일 I/O로 자체 데이터 파일" 방식을 권장한 것과 같은 방향).
